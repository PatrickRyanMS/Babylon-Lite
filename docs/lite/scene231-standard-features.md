# Scene 231 — Standard Material Features (skeleton / morph / vertex color / UV offset)

Reserved scene number **231** (Standard-material skeleton, morph targets, vertex color, UV offset).
This is **new Lite feature work** — there is NO Babylon.js reference to diff against, so the scene
generates its OWN ground-truth golden (like the other Lite-native scenes), and the parity test diffs
future renders against that committed golden to catch regressions.

Belongs on branch **`standard-material-features`** (it exercises that branch's features, and resolves
the "Branch A features are present-but-unexercised" gap).

## Test asset (generated)

`lab/public/test-assets/scene231-standard-features.glb` (34 KB) — generated by
`scratch/scene231-gen/gen.mjs` (uses `@gltf-transform/core`). A single tube mesh (359 verts, 680
tris) carrying ALL four features' data in one mesh:

- **SKELETON** — 2 joints (`bone0` root, `bone1` at y=1.25). Skin weights ramp 0→1 over y∈[0.6,1.9].
  `bone1` is **posed** (rotated 40° about Z) in the glb, so the tube bends on load — a deterministic
  static pose (no animation needed for the golden).
- **MORPH** — one target (radial bulge ∝ sin(v·π), max at mid-height). Mesh **default weight = 1.0**,
  so the bulge is applied on load.
- **VERTEX COLOR** — `COLOR_0` RGBA gradient (red base → blue tip).
- **UV** — cylindrical `TEXCOORD_0`; the scene's Standard material applies a non-zero `uvOffset`.

(Stored in-repo for now; will move to the Assets repo/CDN like the FBX models, served via a `/test-assets/` path.)

## Scene design (`lab/lite/src/lite/scene231.ts`)

1. `loadGltf(engine, "/test-assets/scene231-standard-features.glb")` → meshes with `.skeleton`,
   `.morphTargets`, `_gpu.colorBuffer` (the glTF loader sets these material-agnostically; the shared
   `_computeMeshFeatures` reads them for any material family).
2. Call the Standard opt-ins so the (net-neutral) feature dispatch is installed:
   `enableStandardSkeleton()`, `enableStandardMorph()`, `enableStandardVertexColor()`,
   `enableStandardUvOffset()` — these must be **exported from `index.ts`** (currently internal).
3. Replace each loaded mesh's material with `createStandardMaterial()` + a small checker/grid diffuse
   texture + `mat.uvOffset = [0.25, 0.1]` (so the offset is visible).
4. Fixed capture camera + a deterministic light; `?capture=1` for the golden render (mirrors scene230).

## ⚠ KEY FINDING — vertex-color stride mismatch (needs a decision)

The glTF loader normalizes `COLOR_0` to a **tight float32x4 RGBA (stride 16)** buffer — the engine-wide
vertex-color layout (PBR, procedural meshes, node/shader materials all bind `float32x4`). But the
**Standard `stdVertexColorExt` I ported for FBX declares `float32x3` (stride 12)** (FBX produces tight
RGB). Binding a stride-16 buffer to a stride-12 attribute misaligns every vertex → garbage colors.

So skeleton + morph + UV offset work through the glb→Standard path (shared structures), but **vertex
color does NOT** without a fix. Recommended resolution (a small follow-up, NOT done tonight because it
spans both branches + needs m02 re-validation):

- **Unify Standard vertex color to the engine-standard `float32x4` RGBA** (read `color.rgb`, keep the
  multiply-into-baseColor behavior; optionally let alpha modulate fragment alpha like PBR). Then update
  the FBX loader (Branch B) to emit RGBA instead of tight RGB, and re-confirm m02 (FBX n-gons vcolor).
  After that, scene231 tests all four features through the glb cleanly.

Until then, scene231 should test **skeleton + morph + UV offset** and leave vertex color gated behind a
TODO referencing this doc (or land it together with the vcolor unification).
